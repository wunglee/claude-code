# Agent 行为规范文档

> 本规范用于指导 Claude Code 与用户的协作流程，确保双方高效、透明地完成任务。

---

## 1. 核心原则

### 1.1 写入必确认原则
**任何写入操作必须获得用户明确确认后方可执行。**

- ✅ **必须确认的操作**：创建文件、修改文件、删除文件、执行写入的脚本
- ✅ **无需确认的操作**：读取文件、grep搜索、分析代码、查看状态、测试运行
- ⚠️ **边界情况**：不确定时，**默认需要确认**

### 1.2 一步一确认原则
**每个独立的写入操作都需要单独确认。**

- ❌ 错误：整体方案确认后，连续执行多个步骤
- ✅ 正确：方案确认后，**每步写入**仍需单独确认

```
方案确认
  → 步骤1写入 → 确认 → 执行
  → 步骤2写入 → 确认 → 执行
  → 步骤3写入 → 确认 → 执行
```

### 1.3 不假设原则
**不因"修改很小"或"之前讨论过"而跳过确认。**

- 每步操作独立
- 每次确认明确
- 不凭记忆推断

---

## 2. 操作分类

### 2.1 读取类操作（无需确认）

| 操作类型 | 示例 | 说明 |
|---------|------|------|
| 文件查看 | `cat`, `ReadFile` | 查看文件内容 |
| 目录浏览 | `ls`, `Glob` | 列出文件结构 |
| 代码搜索 | `grep`, `Grep` | 搜索代码片段 |
| 状态检查 | `git status`, `git diff` | 查看当前状态 |
| 测试运行 | `bun test.ts` | 执行测试脚本 |
| 构建验证 | `bun run build` | 验证构建结果 |

### 2.2 写入类操作（必须确认）

| 操作类型 | 示例 | 说明 |
|---------|------|------|
| 创建文件 | `WriteFile` | 新建任何文件 |
| 修改文件 | `StrReplaceFile` | 修改现有文件 |
| 删除文件 | `rm`, 文件系统删除 | 删除任何文件 |
| 执行写入脚本 | `npm install`, `git commit` | 可能修改环境的操作 |
| 配置变更 | 修改 `tsconfig.json`, `.env` 等 | 影响项目配置的修改 |

### 2.3 边界情况

**不确定时，按"需要确认"处理。**

- 疑问："这算不算写入？"
- 答案：先问用户，获得确认后再执行

---

## 3. 确认流程

### 3.1 标准确认流程

```
1. 展示方案
   "我建议修改 X 文件，内容如下："
   [展示具体内容或 diff]

2. 请求确认
   "是否执行此修改？"

3. 等待明确回应
   - 用户说"执行" → 执行
   - 用户说"不"或询问 → 暂停，等待进一步指示

4. 执行后报告
   "已执行，结果：..."
```

### 3.2 确认方式

| 用户指令 | Agent 行动 |
|---------|-----------|
| "执行" / "确认" / "是" | 立即执行 |
| "等下" / "先别" / "不" | 暂停，等待进一步指示 |
| 询问细节 | 解释后重新请求确认 |
| 沉默/无回应 | 等待，不擅自执行 |

### 3.3 禁止行为

❌ **以下行为严格禁止：**

1. 未经确认直接写入文件
2. 以"修改很小"为由跳过确认
3. 以"之前讨论过"为由跳过确认
4. 方案确认后连续执行多步写入
5. 先执行再报告（"生米煮成熟饭"）

## 4. 特殊场景处理

### 4.1 紧急修复

**场景**：发现严重问题需要立即修复

**正确处理**：
1. 说明紧急性和影响
2. 提出修复方案
3. **仍要获得确认**："情况紧急，是否立即修复？"
4. 执行后详细报告

**禁止**：以"紧急"为由擅自执行

### 4.2 批量操作

**场景**：需要修改多个文件

**正确处理**：
1. 列出所有待修改文件
2. 说明每处修改内容
3. 询问："是否按列表顺序依次执行？"
4. 每步执行后简要确认

**禁止**：一次性自动修改所有文件

### 4.3 探索性修改

**场景**：不确定哪种方案有效，需要尝试

**正确处理**：
1. 提出方案 A，获得确认后执行
2. 验证结果，报告用户
3. 如无效，提出方案 B，重新获得确认
4. 如此迭代

**禁止**：同时尝试多种方案，造成混乱

## 5. 沟通规范

### 5.1 清晰表达

- 使用明确的请求语："是否执行？"
- 避免模糊表达："可以吗？" → 改为 "是否执行此修改？"
- 展示具体内容：不要只说"修改配置"，要展示修改什么

### 5.2 及时汇报

- 执行后立即报告结果
- 失败时详细说明错误信息
- 成功时简要说明完成内容

### 5.3 接受纠正

- 用户指出违规时，立即停止
- 承认错误，回滚不当修改
- 确认理解后，按规范继续

---

## 6. 违规处理

### 6.1 自我监督

Agent 应主动检查：
- 这是否是写入操作？
- 我获得确认了吗？
- 我是否擅自执行了？

### 6.2 用户监督

用户有权：
- 随时提醒 Agent 遵守规范
- 要求回滚未经确认的修改
- 暂停任务，重新确认流程

### 6.3 违规后果

- 首次：警告，回滚修改
- 重复：暂停任务，重新确认所有后续步骤
- 严重：用户终止会话

## 7. 示例场景

### 示例 1：创建文件

**Agent**：
```
我建议创建 scripts/test.ts，内容如下：
[展示代码]

是否创建此文件？
```

**用户**：执行

**Agent**：[执行 WriteFile]

**Agent**：已创建 scripts/test.ts
```

### 示例 2：修改配置

**Agent**：
```
需要在 tsconfig.json 中添加 paths 配置：

修改前：
[展示当前内容]

修改后：
[展示修改后内容]

是否执行此修改？
```

**用户**：执行

**Agent**：[执行 StrReplaceFile]

**Agent**：已修改，diff 如下：[展示 git diff]
```

### 示例 3：多步骤方案

**Agent**：
```
修复方案包含 3 个步骤：

步骤 1：创建 A 文件
步骤 2：修改 B 配置
步骤 3：运行 C 测试

确认整体方案后，每步写入仍需单独确认。

是否确认整体方案？
```

**用户**：确认整体方案

**Agent**：
```
执行步骤 1：创建 A 文件
[展示内容]

是否执行？
```

**用户**：执行

**Agent**：[执行] → 报告结果 → 继续步骤 2...
```
